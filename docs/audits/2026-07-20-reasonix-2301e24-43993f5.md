## 上游问题

插件 Skill 可以保留 raw MCP 名、去前缀后的短名或 Claude plugin-qualified 名，但 Provider 只看到
Reasonix canonical `mcp__<server>__<tool>` 名称。缺少 runtime binding 时会出现两类风险：有效 Skill
调用失败，或为兼容短名而错误扩大子代理权限。上游修复同时要求：

- exact canonical name 优先；portable/raw/visible/Claude-style 名称只在唯一时解析；
- 歧义别名拒绝执行并列出 canonical candidates；
- 别名不进入 Provider schema，不改变工具数量、顺序或 prompt cache 前缀；
- 插件 package provenance 贯穿 live/cached MCP 与 Skill root；
- 权限、Plan Mode、Hook、Evidence 使用 canonical security identity；
- host-only runtime binding 不能被 Skill Markdown 伪造，也不能进入稳定 Skill index；
- live registry 覆盖 cache，cache 只有在当前 Spec fingerprint 匹配时才可提供延迟 binding。

## Reames 采用

### Canonical Registry

`internal/tool.Registry` 新增 `MCPBinding`、`MCPBindings` 与 `ResolveCall`。接受 raw、visible、
`server/tool`、`mcp-tool:server/raw`、portable `mcp__server__tool` 和 Claude plugin-qualified 引用，
但只有唯一匹配才返回实际工具；exact registered name 永远优先。Provider `Schemas()` 仍只导出 canonical
工具并保持原有排序与 canonical schema bytes。

### Plugin provenance 与 cache

live `remoteTool` 和 deferred `lazyTool` 同时保存 raw、visible 与 package owner。Reames 没有复制上游新增的
重复 `Spec.Package` 字段，而是继续使用已有、更强的 `Spec.PackagePolicy.Owner` 作为 package 来源真相；
`plugin.ModelToolName` 只导出 canonical name 计算。`skillMCPBindings` 先读取 live registry，任一 live server
都会压过旧 cache；未连接 server 只有 `LoadCachedSchema(name, SpecFingerprint(spec))` 命中时才生成 binding。

### Plugin Skill runtime binding

`Config.PluginPackageSkillOwners` 保存 canonical skill root 到 installed package owner 的 runtime-only 映射并
返回 defensive copy。`skill.Store` 只在 root 恰有一个 owner 时赋予 `Skill.Plugin`；多 owner root 保持
unowned，不能从任一 package 继承 MCP。

`Store.Prepare` 在调用时追加 host-generated `Runtime MCP tool bindings`，并用不可由 Markdown 构造的
私有状态保证幂等。它把唯一 portable/Claude allowed-tools 映射为 canonical name 与稳定 capability ID；
歧义 literal 不授权任何工具，显式 wildcard 保留原边界并只补不能直接匹配的 canonical MCP 名。`List` 与
`IndexBlock` 不调用 Prepare，因此 runtime 内容不进入 cache-stable prompt index。

`run_skill`、`read_only_skill`、`read_skill`、dedicated subagent Skill、Controller `/skill`、boot slash schema
和 reload 后 slash schema 均在执行或渲染前 Prepare。

### Agent 与子代理安全身份

Agent dispatch/read-only/preview/parallel/memory-compiler lookup 支持唯一 alias；真正的权限、fresh/structured
approval、Plan Mode trust、Pre/PostToolUse Hook、Evidence receipt、subagent effect intent 和 repeat-success
identity 全部使用 canonical name。Provider transcript 与 UI event 可继续显示模型原始调用名，避免篡改上游
对话记录。

`FilterRegistry` 和 `ReadOnlySubagentToolRegistryForDepth` 对 literal 使用 `ResolveCall`，对显式 pattern 只在
canonical provider names 上展开，并在 exclusion/read-only/receipt 边界后加入子 registry。歧义短名不加入。
read-only 子代理仍拒绝无 identity-bound reader receipt 的 MCP 工具。

## 明确差异和未冒充能力

- Reames 当前没有 Reasonix 的 `use_capability` MCP proxy，因此没有复制 restricted proxy 包装器，也不声称
  cache-only/on-demand MCP 能在子代理内绕过直接 registry 可用性；capability ID 只解析到已注册的直接工具。
- Reames 当前插件 Skill 的用户可见 slash 目录尚未实现 Reasonix 的 package-qualified collision UX；本批对
  同一 root 多 owner 采取 fail-closed unowned，而不是任选 package。完整 qualified Skill UX 仍属于 P9。
- 没有放宽 Reames 的 MCP identity receipt、destructive fresh approval、package process sandbox、writer
  worktree、evidence 或 Safe Mode 边界。
