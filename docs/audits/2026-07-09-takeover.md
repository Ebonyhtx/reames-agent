# Reames Agent 接管审计（2026-07-09）

## 结论

Reames Agent 已有一套成熟、测试密度较高的 Agent 内核，不是原型；但仓库治理和架构边界落后于功能实现。当前最准确的定位是：

> **Reasonix 衍生的 Go Agent 内核与桌面产品已经可构建、可全测，但“统一 Controller 边界”、干净主仓库、真实 API 端到端和可重复发布仍未闭环。**

本轮以代码、构建和完整测试为准，不以路线图中的“已完成”标签为准。

## 第一性原理验收框架

一个可长期维护的本地 AI 编程助手至少要同时满足六个条件：

1. **用户闭环**：用户能创建会话、提交任务、审批副作用、看到证据并恢复现场。
2. **单一控制面**：CLI、Desktop、Serve、Gateway 只表达命令和事件，不各自重做运行时语义。
3. **副作用可控**：工具调用经过权限、沙箱、路径约束、密钥脱敏和审计。
4. **状态可恢复**：会话、租约、检查点、后台任务和崩溃恢复具有明确所有权。
5. **模型输入稳定**：system prompt、tool schema 和缓存前缀稳定，UI/渠道元数据不污染模型输入。
6. **发布可重复**：干净 clone 上，核心、桌面、前端、资产和安装器都经过自动门禁。

当前内核在 3、4、5 上已有较深实现；1 尚缺真实 API/桌面点击证据；2 是文档目标而非代码事实；6 在本轮前存在明显缺口。

## 仓库事实

- Git 工作主线：`main`，审计开始时工作树干净。
- 跟踪文件约 5,394 个。
- `internal/`：353 个生产 Go 文件、404 个 Go 测试文件。
- `desktop/`：67 个生产 Go 文件、66 个 Go 测试文件；它是独立 Go module，根目录 `go test ./...` 不覆盖它。
- `desktop/frontend/src/`：约 210 个源码文件。
- 仓库最大区域不是新 Go 内核，而是旧 Python/Hermes 资产：`tests/` 约 1,450 文件，另有 `apps/`、`skills/`、`ui-tui/`、`agent/`、`tools/` 等大批遗留目录。
- AGENTS 中“每个 internal package 都有 package comment”的描述不准确；审计只找到约 53 处 package comment，而 internal package 路径更多。

## 架构判断

真实主链路是：

```text
Frontend / CLI / HTTP / IM
        │
        ├── 大部分运行命令与事件 ──> control.Controller ──> agent ──> provider/tools
        │
        └── 装配、历史、标题、配置等辅助路径 ──> agent/provider/tool（直接依赖）
```

因此，`control.Controller` 已是主要运行控制面，但文档中的“界面层禁止 import agent/provider/tool”铁律尚未实现。Desktop、CLI、Serve、Bot 的生产代码都存在直接依赖。短期不应做大爆炸重构；应先定义可测的 public boundary，再逐入口收口。

## 本轮发现并修复

1. OpenAI 流中断测试使用 TCP RST，RST 可能丢弃刚写出的 token，导致“输出后禁止重放”测试随机失败。改为确定性的 incomplete SSE，并连续运行 50 次。
2. 前端无法识别后端错误中的 `Reames Agent`（空格）与 `reames-agent`（连字符）差异，租约冲突退化成英文原始错误。已统一识别。
3. Desktop 完整测试此前未进入 CI，实际有多组失败：Windows 路径大小写、用户目录隔离、TOML 路径转义、时间分辨率和品牌迁移契约均已修正。
4. 外部目录引用实现仍输出旧的 `__reamesAgent_external_folder`，与新契约不一致；已统一为 `__reames_agent_external_folder`。
5. Windows NSIS 仍有 Reasonix 宏，update helper 名称与测试漂移；已统一 Reames Agent 命名。
6. PNG/ICNS/ICO 在品牌替换时被当作文本破坏。已从矢量源重新生成三平台资产，并让 Windows ICO 进入版本控制。
7. `cache-impact` workflow 仍监听旧 `main-v2`，已改为 `main`。
8. 主 CI 原先完全不测 Desktop module 和 frontend；已新增 Windows Desktop 全测，以及前端锁文件安装、全测和 production build。
9. 本地 baseline 在缺少前端依赖时仍打印“passed”，且依赖存在时只提示、不构建；现改为明确区分，并在依赖存在时实际构建。
10. 浏览器预览发现左上角仍是固化在 SVG 路径中的 Reasonix 字标；已替换为 Reames Agent 字标并增加 bundle 契约。

## 当前验证基线

本轮本地通过：

```text
go build ./...
go vet ./...
go test ./internal/... -count=1 -timeout 300s
cd desktop && go test . -count=1 -timeout 300s
cd desktop/frontend && corepack pnpm test:all
cd desktop/frontend && corepack pnpm build
go test ./internal/provider/openai -run TestStreamDoesNotReplayAfterOutput -count=50
```

前端预览已实际打开，页面标题、中文会话树、输入区和 Reames Agent 字标可见，控制台无 warning/error。前端 production build 仍提示主 JS/CSS chunk 偏大，这是性能债务，不是构建失败。

## 未闭环风险

### P0

1. **真实 API Key 端到端**：尚未用真实供应商完成“桌面输入 → 流式输出 → 工具审批 → 文件变更 → 会话恢复”。
2. **新 CI 尚未由远端运行证明**：本地已全绿，合入后仍需观察 GitHub Actions，尤其 Windows Desktop job。
3. **Wails 真实交互证据**：浏览器预览已验证渲染与品牌；仍需启动原生 Wails，完成桥接相关核心点击路径并留截图。
4. **干净 clone 发布验证**：本轮补齐了被忽略的 ICO，但仍需由 release workflow 在各原生 runner 上验证安装包。

### P1

1. **Public boundary 收口**：为 Desktop/CLI/Serve/Gateway 建立依赖守卫，逐步消除对 agent/provider/tool 的直接穿透。
2. **Legacy 仓库收敛**：旧 Python/Hermes 文件数量大于新主产品，影响搜索、依赖治理、许可证判断和新贡献者认知。
3. **文档真实性**：多处“全部覆盖/已完成”只是接口名对照，不是行为级 E2E 证据。
4. **前端体积**：主 JS 约 1.09 MB、主 CSS 约 625 KB（未 gzip），需要按用户路径继续拆包。
5. **包职责注释**：补齐 internal package comment，并用自动检查替代文档约定。

## 接管顺序

1. 合入并观察本轮 CI 门禁。
2. 用真实 API 完成一次 Desktop E2E，记录失败分类、缓存命中和恢复行为。
3. 启动 Wails，验证新建会话、发送/停止、模型设置、审批、文件变更和会话恢复。
4. 定义最小 public boundary 与依赖守卫测试，不先搬代码。
5. 按“无引用、无发布依赖、有替代实现”三个条件分批清理 legacy。
6. 再推进 Server/Web/Cloud 与 Gateway 的真实渠道验证。
